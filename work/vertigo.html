Pivot

Product Design

Tools Used

  • Balsamiq
  • Illustrator

Overview

As a member of Berkeley Innovation, a human-centered design club at UC Berkeley, I’m currently working with three other students on a project with Vertigo Software (recently acquired by MLB). We were essentially assigned to design and prototype Git for designers. A list of Git-equivalent features, including branching and the ability to resolve merge conflicts, were given to us, but the design and function of the application was largely up to us.

Preliminary Research

Ian Birnam, Creative Developer at Vertigo and our main contact, laid out a timeline for our semester-long project. During our first week, he had us do some preliminary research, including competitive analysis and user interviews. Nickki, the only one of the four of us who had no experience using Git, took some time to familiarize herself with Git and note any difficulties she had using it for the first time; Ji-hern did some competitive analysis and identified Pixelapse as our main competitor; and Kylon and I conducted user interviews on both designers and users of Git. At the end of the week, we met up and decided on the MVP: a web application that allows designers to work collaboratively on shared files in a design project by being able to track a design file’s history, maintain communication with other designs through comments, and resolve any “merge conflicts" without having to manually edit the file in Illustrator, Sketch, etc.

Initial Sketches

During week two, we jumped right into sketching out an initial design of our web application. Getting started on sketching out an application with so many tasks was pretty daunting, but once we began drawing out our ideas on whiteboard, it was easy for us to lay down some screens without feeling too committed (even sketching with paper and pen can feel too permanent sometimes).

Once we had the basic workflow outlined, we divvied up drawing out the sketches with pen and paper to present to Ian.

Wireframing with Balsamiq

For week three, Ian wanted some higher (but still low) fidelity mockups, so we each drafted up our screen sketches in Balsamiq. This allowed us to really hash out the workflow of our application as well as discover any holes in the workflow. We also made some new decisions about which features would be included in the product. One of these features was the eqivalent of branches in git, which was one of the features that Ian had wanted us to implement in our application. We decided against including branches in our applcation, concluding that it’s unlikely that designers would need a separate set of all the files, especially if they wanted to experiment on just one file.

Another feature we wrangled with was resolving a merge conflict. (If you’re not familiar with Git, a ‘merge conflict’ is basically a fancy term for when your version of a file and another person’s version conflict because you edited the same parts of the file.) An idea that I suggested was to present the user with a list of attributes of each layer (e.g. position, color, stroke width, etc.) that conflicted between the two versions. The user would then be able to pick and choose the attributes from whichever version they wanted to appear in the resolved file. We decided against this, coming to the conclusion that designers don’t usually work in this manner; they would have to talk to their collaborators to figure out a resolution and then manually fix the conflicts with the design software that they’re using. So we decided to simply show the conflicting versions side by side and allow for comments on the conflict page for designers to sort it out on their own.

Mid-Fidelity Mockups

The feedback we got from Ian on our Balsamiq prototype was generally positive! Our solution for merge conflicts, however, still needed some work. He stressed that having a tool to help designers resolve merge conflicts easily and efficiently without having to open it up in the design program would really make the process less painful for designers. So we resurrected my idea for letting the user pick and choose the attributes of the layers that conflicted between the two versions using our web application. Instead of having the user select individual attributes of a layer, however, we decided that selecting an entire layer (or ‘object’ in Illustrator terms) would be less tedious. So for our next iteration of higher fidelity mockups, we translated our Balsamiq sketches to Illustrator mockups using a simple style guide that Ji-hern drafted up, and I came up with this solution for resolving merge conflicts:

Upon arriving at the merge conflict page of a file, users are presented with a preview of the current version of the file as well as the proposed, conflicting revision on the right-hand side for a quick comparison of how the two versions differ. On the left-hand side is where the user can reconcile the differences by choosing which version of the conflicting layers they want to be in the most updated version of the file. The user is shown a live preview of what the new version would look like with those chosen versions of the conflicting layers.

Higher Fidelity Mockups

The next week our team worked on moving our designs up in fidelity so we could present our work to some more folks at Vertigo, including Creative Developer Cheryl MacDonald, Visual Designer Liliya Kriulina (a former DeCal student of mine!), and Creative Developer Sydnee Stanley. At this point everyone on the team had pretty much already consolidated who was in charge of which screens, so I continued developing the merge conflict page as well as the page for a single file revision that shows a side-by-side comparison two successive revisions of a file and the page on which the user enters a comment (equivalent to a Git commit message) on each of the new revisions being uploaded.


After the user uploads any number of files, he is prompted to write a comment for each revision.


The page for a single file revision shows a side-by-side comparison of the revision and the revision that preceded it.


Merge conflict page; the user selects which version of the layers should be kept in the file. A live preview of what the file would look like with the selected layers is shown to the user. If the conflict can’t be solved entirely with this tool, the user has the ability to download the file with the changes that have been made so far, make the necessary edits in the respective design software, and upload it as the new resolved revision.

From this point on, Ian wanted us to use our remaining four weeks to finalize the visual design of our app, so for the next week I worked on further developing the merge conflict page (see above). Sydnee pointed out that the ability to resolve the conflict manually by downloading the file, making the changes in the design software, and reuploading the new revision was unclear since the download button and upload buttons were nowhere near each other and thus had no clear indication that you would follow those steps in that order. So I played around with the interface, initially trying to arrange the buttons in a way that would be more indicative to the order of each subtask. But I could not, for the life of me, get this to work with the current layout I was working with! So rather than being stuck staring at the same screen, I played around the interface and eventually came up with something I liked a lot better:

I changed the new merge conflict page to make the content look less static and more like an interactive tool, adding zoom in/out functionality so that the user may inspect any smaller changes in the file. I also moved the list of layers to be adjacent to the file preview and made it scrollable so that the user will always see the affected changes of selecting a layer rather than having to scroll up and down the page to preview the new version of the file he is trying to consolidate. I also changed the "Download" button to explicitly say "Open in Illustrator" (or Sketch/Photoshop/etc.) depending on the type of file). This gives the user a better indication of why he would need to download the file in the first place. Because of this hint, I decided that it was okay to leave the button overlayed in the corner of the image preview; while the words on the button implicitly explain what to do with file, the placement of the button over the image suggests what the user is downloading (the previewed version that is shown).
